Objective:
Build a patient-trial matching prototype project for clinical trial.

Background:
Clinical trial recruitment is often slow because trial coordinators must manually review eligibility criteria and compare them against patient information. Many eligibility criteria are written in free text, which makes them difficult to operationalize at scale. For this project, focus specifically on the use case of building a patient-trial matching prototype.

Datasets:
Included in the "dataset" folder are:
- trials_clean.csv: cleaned trial-level dataset. This dataset contains clinical trial eligibility criteria, including inclusion and exclusion rules written in natural language.
- eligibility_criteria_chunks.csv: eligibility criteria split into smaller text chunks.
- raw/: original downloaded data before final cleaning.
- statistics/: summary files describing the dataset.
- The file "patient-profiles.jsonl" will be used to run the matching process.

Approach:
The high-level approach would be based on this:
Step 1 — Data Enrichment: If needed pre-process the dataset (trials_clean.csv) to add new attributes, and generate structural JSON (as described more below). I want to follow a Structured Retrieval rather than pure RAG.
Step 2 — Retrieval: Chunk and embed eligibility criteria into a vector store (plan to use LanceDB to get support for both metadata filtering and vector similarity). At query time, embed the patient profile to retrieve the top-K semantically relevant trials via cosine similarity.
Step 3 — Criterion-Level Assessment: For each candidate trial, drill down into its specific criteria using eligibility_criteria_chunks.csv. Pass each inclusion and exclusion criterion individually to an LLM, which will return a structured result: met, not_met, unknown, or missing_info.
Step 4 — Aggregation: Aggregate the individual criterion results into a final trial-level decision (likely_eligible, not_eligible, or needs_review) and generate a ranked eligibility score.

You may use LLMs, RAG, embeddings, traditional NLP models, rules, or a hybrid approach.

Your prototype should ideally demonstrate how a patient profile can be compared against trial eligibility criteria and produce an output such as:
- likely eligible / not eligible / needs review
- eligibility score or ranking
- matched inclusion criteria
- matched exclusion criteria
- missing information
